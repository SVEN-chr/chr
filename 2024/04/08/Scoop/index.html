title:Scoop

⚠️ 提示: 请确保你在阅读本文之前已阅读本指南的前言部分（不需要完整阅读）。
✨ 提示: 为降低门槛，本文尽可能编写得初学者友好，以确保大多数目标读者能够读懂。这可能令有经验的读者感到过分啰嗦，请谅解，在本指南的正文部分中便不会这么啰嗦了。

作为 Windows 用户，你可能已经习惯于在网络上搜索软件的安装包并手动下载安装它们了——然而 Linux 用户一般不会这么做，他们会使用一个系统自带的“**包管理器 (Package Manger)**”来安装软件，通常是程序员常用的那些，例如 Python 和 Git. 包管理器通常需要通过终端调用——在摒弃了对终端的恐惧后，你会意识到这其实很方便，比人工寻找软件的安装包来安装要快捷得多，也更容易管理你安装的软件。

✨ 提示: “包管理器”一词还有很多引申含义，要完整解释它并不容易。稍后，你可以阅读一下本指南基本常识扫盲部分对“包管理器”概念的解释这一小节，以更详细地了解它——如果你在编程领域还只是个新手，暂时可以不要着急点进去阅读，其中的内容可能把你绕晕。

为了方便管理，本指南也会要求你安装一个包管理器——对于 Windows 用户，本指南推荐使用的包管理器是 Scoop. 本指南的许多文章都会提示你通过 Scoop 安装一些必要的软件——当然，你也可以选择使用其他包管理器或干脆不使用包管理器而选择手动安装它们，这是你的自由，但本指南不会帮你解决由此引发的其他问题。

为什么选择 Scoop？

与 macOS 不同——那里的人几乎都用 Homebrew 作为包管理器——在 Windows 上流行的包管理器不少，并且各自都有不少人在使用。流行的主要有三个，WinGet、Chocolatey 以及 Scoop.

WinGet 是 Windows 当前自带的一个包管理器，对于大多数普通用户它非常方便，你可以在终端中运行 winget upgrade --all 轻松升级你电脑上安装的所有软件（事实上我也很推荐使用它，使用 WinGet 与使用 Scoop 并不冲突）。然而 WinGet 本质上只是对各种软件的安装包进行管理，它为你查找应用程序，然后像你平常手动点击安装包安装应用程序一样，把它们装到它们通常所在的位置——再之后便不会做更多事情了。

而 Scoop 做得更多，它更接近于一个“包管理器”，它尝试提取应用程序本身并尽量将它们安装到一个集中的位置（以方便管理），并且可以在安装前后运行自定义脚本以提供更大的灵活性、以及处理环境变量相关的设置。当然，WinGet 这些年也从 Scoop 借鉴了很多，然而仍旧不如 Scoop 强大。

Chocolatey 是另一个十分流行的包管理器，它的用户很多，得益于 Chocolatey 自身庞大的软件库。然而相比 Scoop，Chocolatey 一定程度上存在软件库质量差、环境变量与路径污染严重等问题。许多人因为 Chocolatey 具有更大的软件库而选择使用它而不选择使用 Scoop，然而 Scoop 要添加更多软件也十分简单——Scoop 自身较小的软件库是为了维持其高质量，但这并不妨碍你轻松引入其他用户定义的软件库，本文会在之后教会你怎么做。

关于更多 Scoop 与 Chocolatey 和 WinGet 的比较，你可以参考 Scoop 官方文档的相关部分。

“钉住”版本

最后

Scoop 是一个相当有用的 Windows 包管理工具。如果你经常需要与编程领域的东西打交道，使用 Scoop 这样的工具集中管理它们无疑是更干净与更方便的选择。更重要的是，包管理器通常都提供了像 $ scoop update * 这样的“一键更新”功能，这要比你手动去下载安装包并一个个更新要方便得多。

我尤其喜爱 Scoop 安装多个版本的软件并通过 reset 切换的能力。程序员们常常需要同时与同一个软件的不同版本打交道，相比起每次都卸载重装，通过 Scoop 安装多个版本并通过 reset 切换要方便得多。

我一般习惯于将主要提供命令行工具的软件通过 Scoop 安装——如各种编程语言、它们的包管理工具/构建工具和其他命令行工具。对于 VSCode、JetBrains IDE 这样的软件，我仍倾向于手动通过安装包安装它们——因为许多与这些软件交互的其他软件都假设你将它们安装在特定的位置，使用 Scoop 安装它们可能会导致一些问题。并且它们通常也提供了内置的更新功能，因此我也不需要通过 Scoop 来更新它们，而对于这些软件我也不可能有需要安装多个版本的需求。
